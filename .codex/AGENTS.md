# Codex Configuration

## コア方針

- 常に英語で思考し、日本語で応答する
- 曖昧な指示（例: 「確認して」「調査して」）は、実行前に確認する
- 複数の独立操作がある場合は、可能な限りツールを並列実行する
- ツール結果を評価してから次の手順を決め、必要に応じて計画を更新する

## 回答フォーマット

- 箇条書きを使う場合は数字の箇条書きを使う
- 箇条書きは「大項目の番号 + 配下の番号」で整理する
- 内容に応じて、見出し形式と箇条書き形式を使い分ける
- 同じ論点は分散させず、1か所に集約して説明する

## ワークフロー

- 探索 → 計画 → 実装 → コミットの順で進める
- 変更前に必ず既存コードを読んで理解する
- 実装前に詳細計画を作成する
- 複雑な実装は、実装開始前に計画レビューを行う
- 作業開始前に作業ブランチを作成する
- カレントがデフォルトブランチ以外の場合は、そのまま着手してよいか確認する
- `main` / `develop` への直接コミットは禁止
- 調査やレビューなど、ファイル変更がない作業ではブランチ作成は不要

## 実装原則

- YAGNI: 今必要な機能だけ実装する
- DRY: 重複コードを避ける
- KISS: シンプルさを最優先する

### コメント規約

- コメントは最小限にし、コードで意図を表現する
- コメントは「なぜ」を説明する
- 回避策や一時対処は理由を明記する
- 明白な処理にはコメントしない

## 修正時の注意

- 修正の影響範囲を確認し、既存機能を壊さない
- 影響先で追加修正が必要なら、既存の期待動作を満たすように対応する

## ファイル運用

- GitHub に上げるべきでないファイルは `.gitignore` に追加する

## GitHub Issues / Pull Request

- Issue / PR 操作は `gh` コマンドを使用する
- Read 操作は事前確認不要
- Write / Delete 操作は実施前に必ず確認する
- Issue / PR の作成文は日本語にする

## ルール追加プロセス

ユーザーから恒常運用すべき指示を受けた場合は次を徹底する。

1. 「これを標準のルールにしますか？」と確認する
2. YES を得たら AGENTS.md に追記する
3. 以後は標準ルールとして適用する
